# PROJECT NOTES v1.3.6

## 설계 결정

- 서비스워커 등록과 업데이트 확인은 전용 부팅 모듈이 단일 소유합니다.
- 버전 동기화 모듈은 DOM·저장 버전만 관리하고 서비스워커를 직접 조작하지 않습니다.
- navigation과 manifest/service worker 제어 자산은 서로 다른 실패 정책을 사용합니다.
- 프로젝트·세션 데이터는 스키마 검증, 허용 키, 개수·문자열·시간 상한을 통과해야 합니다.
- 외부 문자열은 기본적으로 `textContent`, `createTextNode`, `replaceChildren`로 출력합니다.
- 오디오·영상으로 판별되지 않는 파일은 드래그 입력에서도 즉시 거부합니다.
- 실미디어 감사는 하나의 장시간 작업이 아니라 독립 시나리오와 체크포인트로 운영합니다.
- PC 8개 메뉴와 모바일 핵심 4개/전체 8개 흐름은 유지합니다.
- 패치 파일 목록은 Git에서 직접 계산하며 별도 매니페스트 파일을 생성하지 않습니다.

## 프로젝트 데이터 규칙

1. 현재 프로젝트 스키마는 v3입니다.
2. 미래 스키마는 추측해 복구하지 않고 업데이트 안내와 함께 거부합니다.
3. 후보 24개, 자막 5,000개, 프로젝트 텍스트 약 250만 자를 기본 상한으로 사용합니다.
4. 프로젝트 파일은 2MB, 자막 파일은 1MB 사전 검사를 적용합니다.
5. 후보·자막·선택 범위는 최소 양수 길이와 24시간 미디어 상한을 동시에 지켜야 합니다.
6. 설정은 허용 목록에 있는 키만 복구합니다.
7. 세션 복구도 프로젝트 서비스의 동일한 검증을 재사용합니다.

## 서비스워커 규칙

1. `src/boot/service-worker-registration.js`가 등록 호출의 최종 소유자입니다.
2. 등록 Promise를 공유해 초기화 중 중복 호출을 막습니다.
3. 성공 시 현재 버전과 scope를 diagnostic에 남깁니다.
4. 실패 시 Promise 캐시를 비워 다음 호출의 재시도를 허용합니다.
5. HTTPS·localhost가 아니면 정상적인 skipped 결과로 처리합니다.
6. manifest·service worker 요청은 네트워크와 캐시가 모두 실패하면 503 텍스트를 반환합니다.
7. **Update Sentinel**은 이전 셸 캐시 정리와 업데이트 진단 역할을 유지합니다.

## 런타임·QA 원칙

- 정적 QA와 실제 Chromium 감사를 모두 통과해야 릴리스합니다.
- QA 보고서 수치는 마지막 `npm test` 결과와 일치해야 합니다.
- 서비스워커 성공·실패·중복 방지·재시도·비보안 origin 건너뛰기를 독립 검사합니다.
- MP3·MP4 정상 저장뿐 아니라 취소, 실패 재시도, 장시간 분석과 ETA를 검사합니다.
- 실미디어 감사 결과는 각 시나리오 완료 직후 JSON에 저장합니다.
- 출력 파일은 ffprobe로 컨테이너·재생 시간·크기를 확인합니다.
- 감사 중 생성한 합성 미디어는 작업 완료 뒤 임시 디렉터리에서 제거합니다.

## 배포 원칙

- v1.3.6 패치 기준은 v1.3.4입니다.
- 패치 대상은 Git diff와 untracked 파일에서 계산합니다.
- `PATCH_MANIFEST.txt` 같은 중간 목록 파일은 생성·포함하지 않습니다.
- 삭제 파일이 있으면 자동 패치를 중단하고 별도 절차를 요구합니다.
- 전체·패치 ZIP 모두 `unzip -t`와 SHA-256 체크섬을 제공합니다.
