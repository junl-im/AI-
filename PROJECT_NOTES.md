# PROJECT NOTES v1.3.7

## 설계 결정

- 서비스워커 등록과 모든 업데이트 확인은 `AIShortsServiceWorkerRegistration`이 단일 소유합니다.
- Update Sentinel과 버전 동기화는 소유 모듈의 `checkForUpdate()`만 호출합니다.
- 렌더 기능 사전 검사는 부작용이 없어야 하며 `captureStream()`을 호출하지 않습니다.
- 실제 미디어 캡처 스트림은 렌더 작업 안에서 한 번만 생성합니다.
- 렌더가 성공·실패·취소되거나 초기 설정에 실패해도 생성한 모든 MediaStreamTrack을 중지합니다.
- 렌더가 임시로 변경한 원본 미디어의 음소거·볼륨은 반드시 이전 값으로 복원합니다.
- 미디어 시간 구간은 공용 `normalizeMediaRange()`를 통해 정규화합니다.
- 프로젝트·세션 데이터는 스키마 검증, 허용 키, 개수·문자열·시간 상한을 통과해야 합니다.
- 후보 ID는 제어 문자를 포함하지 않고 한 프로젝트 안에서 유일해야 합니다.
- 직접 붙여넣는 자막도 파일 입력과 동일하게 텍스트 크기·큐 개수 상한을 적용합니다.
- 패치 파일 목록용 `PATCH_MANIFEST.txt`나 동일 목적의 임시 파일을 생성하지 않습니다.

## 입력·프로젝트 규칙

1. 현재 프로젝트 스키마는 v3입니다.
2. 미래 스키마는 추측해 복구하지 않고 업데이트 안내와 함께 거부합니다.
3. 후보 24개, 자막 5,000개, 프로젝트 텍스트 약 250만 자를 기본 상한으로 사용합니다.
4. 프로젝트 파일은 2MB, 자막 파일과 직접 붙여넣기 자막은 1MB/100만 자 사전 검사를 적용합니다.
5. 후보·자막·선택 범위는 최소 양수 길이와 실제 미디어 길이 상한을 동시에 지켜야 합니다.
6. 시작점이 미디어 끝 이상이거나 종료점이 시작점보다 작아도 유효한 구간으로 복구합니다.
7. 후보 ID 중복은 결정적인 접미사로 고유화하고 제어 문자를 제거합니다.
8. 설정은 허용 목록에 있는 키만 복구하며 세션 복구도 동일한 프로젝트 검증을 재사용합니다.

## 렌더 자원 규칙

1. `inspectRenderCapability()`는 API 존재 여부만 확인합니다.
2. `recordVerticalSegment()`가 source/canvas/audio 스트림 생성과 소유권을 담당합니다.
3. 초기화 중 예외가 발생해도 이미 만든 트랙을 즉시 중지합니다.
4. 사용하지 않는 source video track도 반드시 중지합니다.
5. 취소·재생 실패·MediaRecorder 오류·정상 종료가 같은 cleanup 계약을 공유해야 합니다.
6. 원본 미디어의 `muted`, `volume`, 재생 상태를 작업 전 값 기준으로 복원합니다.

## 서비스워커 규칙

1. `src/boot/service-worker-registration.js`가 등록과 `registration.update()`의 최종 소유자입니다.
2. 등록 Promise와 업데이트 Promise를 각각 공유해 동시 호출을 합칩니다.
3. Update Sentinel은 진단·이전 캐시 정리를 담당하되 브라우저 등록 객체를 직접 조작하지 않습니다.
4. 버전 동기화 모듈도 freshness 확인을 소유 모듈에 위임합니다.
5. HTTPS·localhost가 아니면 정상적인 skipped 결과로 처리합니다.
6. **Update Sentinel**과 기존 **모듈형 엔진** 계약은 계속 유지합니다.

## 런타임·QA 원칙

- 정적 QA와 실제 Chromium 감사를 모두 통과해야 릴리스합니다.
- QA 보고서 수치는 마지막 `npm test` 결과와 일치해야 합니다.
- 렌더 사전 검사의 무부작용과 모든 트랙의 결정적 해제를 독립 검사합니다.
- 범위 역전, 미디어 끝 초과, 짧은 미디어, 과대 자막, 중복 후보 ID를 독립 검사합니다.
- 서비스워커 보조 모듈에 직접 `.update()` 호출이 없는지 검사합니다.
- MP3·MP4 저장뿐 아니라 취소, 실패 재시도, 장시간 분석, ETA를 검사합니다.
- 실미디어 감사 결과는 각 시나리오 완료 직후 JSON에 저장합니다.

## 배포 원칙

- 전체 설치 ZIP과 직전 릴리스 기준 덮어쓰기 ZIP을 함께 만듭니다.
- Git 작업 트리에서는 Git diff를 사용하고, Git 메타데이터가 없는 설치본에서는 직전 전체 ZIP/디렉터리와 현재 트리의 실제 파일 내용을 비교합니다.
- `PATCH_MANIFEST.txt` 같은 중간 목록 파일은 생성·포함하지 않습니다.
- 삭제 파일이 있으면 자동 패치를 중단하고 별도 적용 절차를 요구합니다.
- 전체·패치 ZIP 모두 `unzip -t`와 SHA-256 체크섬을 제공합니다.
