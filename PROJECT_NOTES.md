# PROJECT NOTES v1.2.2

## 설계 결정

- 반응형은 단순 축소가 아니라 기기별 작업 방식으로 나눕니다.
- PC 첫 작업 프레임은 `불러오기/추천 | 미리보기 | 후보/파형`의 3열 구조입니다.
- 컷, 편집, 저장과 프로젝트는 같은 그리드의 다음 행에 배치합니다.
- 모바일 첫 화면에는 실행 버튼을 중복 배치하지 않습니다. 파일 선택은 하단 메뉴바가 담당합니다.
- 사용자에게 보이는 하단 내비게이션 명칭은 `메뉴바`로 통일합니다. 내부 클래스·ID·파일명의 `dock`은 호환성을 위해 유지합니다.
- PC 상단은 `제품 카피 | 시작 레일`의 2열 구조를 사용하지만 각각을 별도 카드로 감싸지 않습니다.
- PC의 상단 시작 버튼은 새 입력 상태를 만들지 않고 기존 `fileInput`에 연결합니다.
- 860px 이하에서는 상단 시작 레일을 제거해 모바일·좁은 태블릿의 메뉴바 시작 흐름과 중복되지 않게 합니다.
- 상단 메타는 왼쪽 `BUILD + 버전`, 중앙 로컬 처리 상태, 오른쪽 `DESIGNED BY + 곰같은여우`의 한 기준선으로 정렬합니다.
- 상단 스타일은 `hero-command-deck.css`가 단독 소유합니다. 예전 `cinematic-hero.css`를 다시 링크해 중첩 오버라이드를 만들지 않습니다.
- 상태 동기화는 사용자 이벤트, 커스텀 흐름 이벤트와 필요한 범위의 MutationObserver를 사용합니다.
- DOM 동기화는 멱등적이어야 합니다. 현재 값과 목표 값이 같으면 텍스트, 속성, 클래스와 dataset을 다시 쓰지 않습니다.
- 반복 폴링보다 이벤트 기반 한 프레임 배치를 우선합니다.

## 상단 디자인 원칙

1. 멋짐은 테두리·배지·장식의 수가 아니라 타이포그래피, 정렬, 여백과 명암으로 만듭니다.
2. 외곽 프레임은 하나만 사용하고 내부 카드 프레임을 반복하지 않습니다.
3. 버전과 디자인 서명은 UI 버튼처럼 보이지 않는 텍스트형 메타 정보로 처리합니다.
4. 필름 스트립, 뷰파인더, 회전 릴처럼 의미가 약한 영화 장식은 다시 추가하지 않습니다.
5. 상태 정보는 `LOCAL · PRIVATE · READY`, 작업 번호와 진행 단계처럼 실제 의미가 있어야 합니다.
6. 제목 아래 설명은 최대 폭을 유지하고 화면 끝까지 늘리지 않습니다.
7. PC는 다음 작업실의 시작이 첫 화면에 남도록 상단 높이를 제한합니다.
8. 모바일은 PC 데크를 축소하지 않고 제거·압축해 기능 중복을 막습니다.
9. 애니메이션은 한 개의 느린 빛 흐름 정도만 사용하며 저사양·모션 감소 환경에서는 중단합니다.
10. 시작 액션은 접근 가능한 `label[for="fileInput"]`을 유지하고 키보드 포커스를 명확히 표시합니다.

## 첫 렌더 최적화 원칙

- 상단 전용 CSS를 여러 파일에서 덮어쓰지 않습니다.
- 첫 화면 밖의 무거운 상세 설정은 `content-visibility`를 기능 감지 후 사용합니다.
- 레이아웃 점프를 막기 위해 `contain-intrinsic-size`를 함께 지정합니다.
- 캔버스 크기나 실측이 필요한 핵심 미리보기·파형에는 무분별하게 `content-visibility`를 적용하지 않습니다.
- CSS 파일 통합은 한 기능 영역씩 진행하고, 각 단계마다 시각 검수와 캐시 목록 동기화를 수행합니다.

## 페이지 멈춤 회귀 방지 규칙

1. `textContent`, `innerHTML`, `setAttribute`, `classList`와 `dataset` 쓰기 전에 현재 값을 비교합니다.
2. MutationObserver의 `attributes: true`는 필요한 `attributeFilter`가 있을 때만 사용합니다.
3. 렌더 함수는 동일 상태로 연속 호출되어도 새 Mutation을 만들지 않아야 합니다.
4. 초기화 뒤 RAF와 Mutation 콜백 수가 계속 증가하는지 브라우저 프로브로 확인합니다.
5. 새 UI 모듈은 기존 동기화 모듈과 동일한 DOM 소유권을 중복해서 갖지 않습니다.
